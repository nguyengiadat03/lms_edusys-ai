# Environment Variables Security Guide

## 🔴 CRITICAL SECURITY WARNING

**NEVER commit real credentials to Git!**

The `.env` file contains sensitive information and should NEVER be committed to version control.

## Setup Instructions

### 1. Copy the Example File

```bash
cp .env.example .env
```

### 2. Update with Your Credentials

Edit `.env` and replace all placeholder values with your actual credentials.

### 3. Verify .gitignore

Ensure `.env` is in `.gitignore`:

```
# .gitignore
.env
.env.local
.env.*.local
```

## Required Environment Variables

### Database Configuration

```bash
DB_HOST=your_database_host          # e.g., localhost or 45.32.100.86
DB_PORT=3306                        # MySQL default port
DB_USERNAME=your_database_user      # Database username
DB_PASSWORD=your_secure_password    # Strong password (min 16 chars)
DB_DATABASE=your_database_name      # Database name
```

**Security Tips:**

- Use strong passwords (min 16 characters, mixed case, numbers, symbols)
- Never use default passwords
- Rotate passwords regularly
- Use different passwords for dev/staging/production

### JWT Secrets

```bash
JWT_SECRET=your_jwt_secret          # Min 32 characters
JWT_REFRESH_SECRET=your_refresh_secret  # Min 32 characters
```

**Generate Strong Secrets:**

```bash
# Generate a random 64-character hex string
node -e "console.log(require('crypto').randomBytes(64).toString('hex'))"

# Or use OpenSSL
openssl rand -hex 64
```

**Security Tips:**

- NEVER use the example secrets in production
- Generate unique secrets for each environment
- Store secrets securely (use secret management tools in production)
- Rotate secrets periodically

### CORS Configuration

```bash
FRONTEND_URL=http://localhost:8080
CORS_ALLOWED_ORIGINS=http://localhost:8080,http://localhost:5173
```

**Security Tips:**

- Only allow trusted origins
- Use specific URLs, not wildcards in production
- Update when deploying to new domains

## Optional Environment Variables

### OCR Service

```bash
OCR_SERVICE_URL=http://localhost:8000
FFMPEG_PATH=/path/to/ffmpeg/bin
WHISPER_ENABLED=true
```

### AI Services

```bash
GEMINI_API_KEY=your_gemini_api_key
AI_SERVICE_KEY=your_openai_key
```

**Security Tips:**

- Keep API keys secret
- Use API key restrictions (IP, domain, etc.)
- Monitor API usage
- Rotate keys if compromised

### Email Configuration

```bash
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=your-email@gmail.com
SMTP_PASS=your-app-password
```

**Security Tips:**

- Use app-specific passwords, not account passwords
- Enable 2FA on email accounts
- Restrict SMTP access by IP if possible

## Environment-Specific Configuration

### Development (.env)

```bash
NODE_ENV=development
LOG_LEVEL=debug
SKIP_DB_TEST=false
```

### Staging (.env.staging)

```bash
NODE_ENV=staging
LOG_LEVEL=info
SKIP_DB_TEST=false
```

### Production (.env.production)

```bash
NODE_ENV=production
LOG_LEVEL=warn
SKIP_DB_TEST=true
```

## Security Checklist

### Before Committing

- [ ] `.env` is in `.gitignore`
- [ ] No real credentials in `.env.example`
- [ ] All secrets are placeholders in example file
- [ ] Documentation is up to date

### Before Deploying

- [ ] Strong JWT secrets generated
- [ ] Database password is strong and unique
- [ ] CORS origins are restricted
- [ ] API keys are valid and restricted
- [ ] Environment variables are set in deployment platform
- [ ] Secrets are stored in secure secret management system

### Regular Maintenance

- [ ] Rotate JWT secrets every 90 days
- [ ] Rotate database passwords every 90 days
- [ ] Review and update CORS origins
- [ ] Audit API key usage
- [ ] Review access logs for suspicious activity

## What to Do If Credentials Are Exposed

### 1. Immediate Actions

1. **Rotate all exposed credentials immediately**
2. **Revoke compromised API keys**
3. **Change database passwords**
4. **Generate new JWT secrets**
5. **Review access logs for unauthorized access**

### 2. Remove from Git History

If credentials were committed to Git:

```bash
# Remove file from Git history (use with caution!)
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch backend/.env" \
  --prune-empty --tag-name-filter cat -- --all

# Force push (WARNING: This rewrites history!)
git push origin --force --all
```

**Better approach:** Consider the repository compromised and:

1. Rotate ALL credentials
2. Create a new repository
3. Copy code (not history) to new repo
4. Update all deployment configurations

### 3. Notify Team

- Inform all team members
- Update deployment configurations
- Document the incident
- Review security practices

## Secret Management Best Practices

### Development

- Use `.env` files (never commit)
- Share secrets securely (1Password, LastPass, etc.)
- Use different credentials than production

### Production

Use a secret management service:

- **AWS Secrets Manager**
- **Azure Key Vault**
- **Google Secret Manager**
- **HashiCorp Vault**
- **Doppler**
- **Infisical**

### CI/CD

- Store secrets in CI/CD platform (GitHub Secrets, GitLab CI/CD Variables)
- Never log secrets
- Use masked variables
- Limit access to secrets
